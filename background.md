# プロジェクト背景（Cursor 用コンテキスト）

このアプリを Cursor で改修・拡張する際の判断基準として参照するドキュメント。  
開発の動機・採用フレームワーク・既存課題・関連プロジェクトを整理する。

---

## 1. 開発の動機

### 開発者本人の状況

- 開発者は日本人で、フィリピンのPines Academy（Baguio）で約5週間の英語留学中。
- 英語学習の主目的：英語を通じてメッセージを的確に理解し、相手にわかりやすく返すこと。
- 強み：論理構造で整理する思考（コンサル出身・MECE志向）。
- 弱み：英語の「お作法（文法・構造）」を体系的に積み上げてこなかったため、聴き取り・読み・書きで構造が崩れる。

### 自覚している3つの具体的な課題

1. **単純な骨格（S+V+O/C）を作れない**  
   主語と動詞を即座に決められず、英作文・スピーキングで詰まる。

2. **句・節のフレームが固まっていない**  
   不定詞・動名詞・分詞・関係詞・副詞節などの「どの形で・どの位置に置けるか」が曖昧。

3. **ネストが深くなると一気に崩れる**  
   例：`People who work with me study hard when some issue occurs.`  
   関係詞節（Y）と副詞節（Z）が骨格（X+V）にぶら下がる構造で詰まる。

このアプリは、上記3つの課題を **「日本語→英語の翻訳」という能動的なアウトプット練習** で潰すことを目的とする。

## 2. 採用する文法フレームワーク（X / Y / Z）

開発者は事前にClaudeとの対話を通じて、英語の文構造を再帰的に捉えるフレームを構築している。Cursor もこのフレームに合わせて拡張すること。

### 基本原理

文の構造は3層しかない。

```
① 骨格（文型）= S + V (+O / +C)
     └ 動詞は4つの情報を背負う：時制・相・態・助動詞
② 骨格の各位置を「かたまり（句・節）」が埋める／飾る
     かたまりは必ず X / Y / Z の3役割のどれか
③ 骨格が複数あるとき → 接続でつなぐ
```

### 3つの役割

- **X：名詞役** … S・O・C になれる。形：名詞 / 動名詞 / 不定詞 / that節 / wh節
- **Y：形容詞役** … 名詞を後ろから飾る（後置修飾）。形：形容詞 / 前置詞句 / 不定詞 / 分詞 / 関係詞節
- **Z：副詞役** … 動詞・文を飾る。形：副詞 / 前置詞句 / 不定詞 / 分詞構文 / 副詞節

### 文型の公式（X/Y/Z 表記）

- S + V + O = `X + V (+Z) + X`
- S + V + O + O = `X + V (+Z) + X + X`
- S + V + C = `X + V (+Z) + (X or Y)`
- S + V + O + C = `X + V (+Z) + X + (X or Y)`

### 再帰性

X・Y・Z のルールは句・節の **内部でも再帰的に適用される**。  
（例：副詞節 `when an issue arises` の内部にも `an issue (X) + arises (V)` がある）

このフレームは、ステップ別の問題生成・採点プロンプト（`src/prompts/index.js`）にも反映済み。

## 3. 学習ステップ設計

ステップは、扱えるかたまりの長さ・入れ子の深さを一段ずつ伸ばすように設計されている。

| Step | 内容 | 課題への効き目 |
|------|------|---------------|
| 1 | 品詞（学習済み） | 構造を見る前提 |
| 2 | 文型・自他動詞（学習済み） | 課題①の基礎 |
| **3** | **動詞の変化（時制・相・態・助動詞）** | 骨格の正確化（課題①） |
| **4** | **準動詞・前置詞句** | 句が3役割に登場する最頻出パーツ（課題②） |
| **5** | **関係詞** | 後置修飾＝日本語との最大の差を崩す（課題③） |
| **6** | **節と接続・ネスト** | 課題③の本丸 |
| 7 | その他（比較・仮定法・倒置） | 後回し |

このアプリは **Step 3〜6** を対象とする。Step 7以降は別途検討。

## 4. UX の設計判断

### なぜ「全問まとめて出題 → まとめて答え合わせ」か

- 1問ずつだと、骨格を意識する間もなく答えを見てしまい、能動性が下がる。
- 5問まとめて全部書ききる → まとめて採点、というリズムが集中を維持しやすい。
- 採点後に「色分け表示・骨格フレーム・各チャンクの役割」を一括で振り返れる。

### なぜヒントを削ったか

- ヒントがあると、思考の起点が「ヒントを見る」になり、自分で構造を組み立てる訓練にならない。
- 失敗してでも自分で骨格を組み、後で構造分解と照合するほうが学習効率が高い（active recall）。

### なぜ自己採点（できた / 要復習）を残したか

- AIの採点が誤判定する場合がある（許容される表現を不正解とするなど）。
- 学習者本人の主観で「身についた／曖昧」を区別できる二段構えにしたい。

## 5. 技術スタック選定理由

- **React + Vite**：開発者は他プロジェクトでもCursor経由でReact開発を行っており、移植しやすい。
- **GitHub Pages**：開発者の他アプリ（English-Questions, English-Phrase）と同じデプロイ方式で統一。
- **Claude API（Haiku 4.5）**：採点・問題生成ともに精度と速度のバランスが良い。コストも低い。
- **APIキーは localStorage**：個人利用前提なので簡素化。`anthropic-dangerous-direct-browser-access` を許容。
- **静的シード問題を残す**：API不調時のフォールバック・初回表示の高速化・プロンプト改修時の参照用。

### 採用しなかった選択肢

- **バックエンドプロキシ（Cloudflare Workers等）**：個人利用なら過剰。後日複数人で使うなら再検討。
- **状態の永続化（IndexedDB等）**：採点結果をまたいで継続学習する設計ではないため不要。

## 7. Cursor が拡張する際の指針

### 守ってほしいこと

- **X / Y / Z の3役割表記はぶらさない**：UI・プロンプト・データ構造すべてで一貫させる。
- **役割の色定義は `src/constants/roles.js` のみで管理**：散在させない。
- **能動アウトプット → 構造の照合という流れを崩さない**：先にヒントを出す・即答を表示するUXは避ける。
- **静的シード問題は削らない**：API不調時のフォールバック・テスト時の参照に必須。

### 改善歓迎

- 採点結果の永続化（要復習だけ抽出する復習モード）
- ステップごとの学習履歴・正答率の可視化
- AI生成問題のキャッシュ（同じステップで再生成不要にする）
- プロンプトの分割管理（Step別に細かく調整できるように）

### 慎重に扱ってほしい

- プロンプトの大幅な作り変えはJSON返却の安定性を壊しやすい。`src/api/claude.js` の `extractJsonArray` も合わせて確認すること。
- モデル変更時は `MODEL` 定数のみ変更で済むようにしている。複数モデルを混在させる場合は構造を見直す。
